Backend Services For iOS Fitness App

About Application

One of the top rankings US based iOS fitness app.

Requirements

Phase 1

Migrate the application backend services, built on Facebook's parse server, to another platform hosting open source parse server. Also, migrate the accompanying mongoDB deployment to a new plan. This incuded both production and development instances.

Phase 2

Development of:
  • Multiple services to dispatch emails depending on the weekly acitivity/inactivity of users, new excercise logging etc. Tweak and improve the email sending logic on regular intervals
  • A service that unsubscribes a user from multiple email providers with one click (mandrill and mailchimp)
  • A minimal portal where admin can 1. Switch on/off any of the above background jobs 2. View aggregated data of production db (no of new users, active users; weekly, monthly etc) in graphical representation 3. View and download logs, generated by the above background services and stored in mongoDB, in CSV file
  • Occasional functions in parse server cloud code

    • Requirement Gathering

    All requirements and conversations generated by them would be put on Trello. The time against task was also logged there. For lengthy conversation Skype was used.

    Resource

    Single

    Implementation

    For phase one the production parse server was deployed on Amazon Elastic Beanstalk. The dev server was hosted on EC2 instance. MongoDBs for both were mirated to an mLab plan.

    In second phase, the backend services were developed with Node, Express and mongoDB (readonly connection to production database and another database inducted for chores). Unit tests written in Mocha. Mandrill the email provider. Angular was used on client side. The deployment was EC2.

    Impact

    The graph of new and active users for the app went many folds up after the integration of activity based weekly emails.

    Duration

    Phase 1

    ~2 weeks

    Phase 2

    Ongoing (I no longer work on it)

    Property Management Panel

    About Application

    A web panel for agency's internal use, mainly to view and process new and updated properties synchronised from multiple RETS (Real Estate Transaction Standard) feeds.

    Technology Stack

    The application was already built when I was inducted. The stack was:
  • Node
  • Express
  • PostgreSql
  • Angular
  • Handlebars (view engine)

    • Requirements

    The requirements and their implementation included:

  • Integration of three RETS feeds (before that records would be imported from simplyRets), automatically (schedulers) and manually
  • Geocode the poperties, both manually and automatically, single and in batches
  • Calculate certain price related averages automactially
  • Upload CSV files containing property data
  • Log the events of above background jobs in the database. Show the logs in a view and make them downloadable
  • Fix/improve the current views that included maps, sorting/filtering functionality, and other such options
  • Entertain queries related to properties in the database

    Through Pivotal Tracker and Skype
